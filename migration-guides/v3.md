## What's new in v3?

The v3 release is a much more modest update than v2. We have collected a small
set of changes for both naming clarity and correctness.

## Add `contentDOMRef` instead of automatically detecting `contentDOM`

This is the biggest change for consumers of this library. In v2, you don't need
to manually specify your `contentDOM`, we just give you children and you render
them.

In v3, we will pass a `contentDOMRef` prop in `nodeProps` that must be assigned
to the element that is the parent of the children prop, if the node view
component renders its children.

Since it's not uncommon to have `dom` and `contentDOM` be the same element, we
also expose a `useMergedDOMRefs` utility hook that allows you to place `ref` and
`contentDOMRef` on the same element.

### To migrate

For every node view component that renders `children`, add
`nodeProps.contentDOMRef` to the parent element of `children`. For example:

```tsx
// If you have this in v2
function Card({ nodeProps, ref, children, ...props }: NodeViewComponentProps) {
  return (
    <div ref={ref} {...props}>
      <div>{children}</div>
    </div>
  );
}

// You need this in v3
function Card({ nodeProps, ref, children, ...props }: NodeViewComponentProps) {
  return (
    <div ref={ref} {...props}>
      {/* Add the contentDOMRef explicitly */}
      <div ref={nodeProps.contentDOMRef}>{children}</div>
    </div>
  );
}
```

If you have any node view components that only have one element, you can use
`useMergedDOMRefs` to set both `ref` and `contentDOMRefs` on that element:

```tsx
// If you have this in v2
function Paragraph({
  nodeProps,
  ref,
  children,
  ...props
}: NodeViewComponentProps) {
  return (
    <p ref={ref} {...props}>
      {children}
    </p>
  );
}

// You need this in v3
import { useMergedDOMRefs } from "@handlewithcare/react-prosemirror";

function Paragraph({
  nodeProps,
  ref,
  children,
  ...props
}: NodeViewComponentProps) {
  return (
    <p ref={useMergedDOMRefs(ref, nodeProps.contentDOMRef)} {...props}>
      {children}
    </p>
  );
}
```

## Rename `nodeView` and `markView` props

In v2, we needed a way to distinguish between React components that React
ProseMirror could use as node views and mark views, and “node view constructor”
functions, the lower-level manually-managed ProseMirror View node view
implementations.

We initially called the prop for the React components `nodeViews` (and
`markViews` for marks), and used `customNodeViews` (and `customMarkViews`) for
the native node view constructors.

Ultimately this was sort of a confusing choice, and it deviated from the
ProseMirror EditorView API without sufficient justification. So in v3, we revert
to using `nodeViews` and `markViews` as ProseMirror defines them (to refer to
the native node view constructors), and use `nodeViewComponents` and
`markViewComponents` to more clearly reference React component node/mark view
implementations.

### To migrate

- Rename your `nodeViews` prop on the `ProseMirror` component to
  `nodeViewComponents`
- Rename your `markViews` prop on the `ProseMirror` component to
  `markViewComponents`
- Rename your `customNodeViews` prop on the `ProseMirror` component to
  `nodeViews`
- Rename your `customMarkViews` prop on the `ProseMirror` component to
  `markViews`

```tsx
// If you have this in v2
const nodeViewComponnets = {
  card: Card,
};

const nodeViewConstructors = {
  paragraph: () => {
    const dom = document.createElement("p");
    return {
      dom,
      contentDOM: dom,
    };
  },
};

function Editor() {
  return (
    <ProseMirror
      defaultState={EditorState.create({ schema, plugins: [reactKeys()] })}
      nodeViews={nodeViewComponents}
      customNodeViews={nodeViewConstructors}
    >
      <ProseMirrorDoc />
    </ProseMirror>
  );
}

// You need this in v3
const nodeViewComponnets = {
  card: Card,
};

const nodeViewConstructors = {
  paragraph: () => {
    const dom = document.createElement("p");
    return {
      dom,
      contentDOM: dom,
    };
  },
};

function Editor() {
  return (
    <ProseMirror
      defaultState={EditorState.create({ schema, plugins: [reactKeys()] })}
      nodeViewComponents={nodeViewComponents}
      nodeViews={nodeViewConstructors}
    >
      <ProseMirrorDoc />
    </ProseMirror>
  );
}
```

## Clean up props

The props for ProseMirror and ProseMirrorDoc were a little too redundant for our
liking. Taking just `className` as an example, you could set `className` on:

1. The `className` prop of the ProseMirror component
2. The `attributes.class` prop of the ProseMirror component
3. The `className` prop of the ProseMirrorDoc component
4. The `className` prop of the element passed to ProseMirrorDoc's `as` prop

4 is… too many. So we've decreased to 2, which is essentially the same number
that ProseMirror supports. You can now set `className` on:

1. The `attributes.class` prop on ProseMirror
2. The `className` prop on ProseMirrorDoc

And we'll properly merge them, too.

### To migrate

- If you set the `className` prop on the `ProseMirror` component, move it to the
  `attributes.class` prop on `ProseMirror` or `className` on `ProseMirrorDoc`
- If you used the `as` prop on `ProseMirrorDoc`, update it to pass the Component
  or element type (e.g., `<article />` becomes `"article"` and `<Article />`
  becomes `Article`)
- If you passed props on the element that was passed to `as`, move them to the
  `attributes` prop on `ProseMirror` or direct props on `ProseMirrorDoc`

```tsx
// If you had this in v2
function Editor() {
  return (
    <ProseMirror
      className="editor"
      defaultState={EditorState.create({ schema, plugins: [reactKeys()] })}
    >
      <ProseMirrorDoc as={<article data-editor />} />
    </ProseMirror>
  );
}

// You need this in v3
function Editor() {
  return (
    <ProseMirror
      defaultState={EditorState.create({ schema, plugins: [reactKeys()] })}
    >
      <ProseMirrorDoc as="article" className="editor" data-editor />
    </ProseMirror>
  );
}
```
